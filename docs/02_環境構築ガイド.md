## 概要

本プロジェクトでは段階リリースを採用します。Phase 1 は Supabase を使わず、Phase 2 以降で開発環境とリリース環境で異なる Supabase 環境を使用します。環境変数は`flutter_dotenv`パッケージで管理します。

## フェーズ別の前提

### Phase 1

- 未ログインのローカル記録版
- Supabase の起動、`.env.local`、Auth 設定は不要

### Phase 2以降

- Google / Apple ログインとクラウド同期を導入
- このガイドの Supabase セットアップを利用する

## 環境構成

| 環境 | 環境変数ファイル | Supabase環境 | 用途 |
|------|------------------|--------------|------|
| 開発 | `.env.local` | ローカル（supabase start） | ローカル開発・テスト |
| リリース | `.env` | リモート（Supabase Cloud） | 本番環境 |

## Phase 1 アプリ内 SQLite DB の確認

Phase 1 の未ログイン記録データは Supabase ではなく、端末内の SQLite ファイルに保存されます。

- DB ファイル名: `base_match_local.sqlite`
- 定義ファイル: `lib/core/local_db/local_database.dart`

iOS シミュレータで VS Code から確認する場合は、アプリを起動した状態で次を実行します。

```bash
code "$(xcrun simctl get_app_container booted com.basematch.app data)/Documents/base_match_local.sqlite"
```

VS Code で開くには、SQLite Viewer 系の拡張機能を入れておくと便利です。

- `SQLite Viewer` (`qwtel.sqlite-viewer`)
- `SQLite` (`alexcvzz.vscode-sqlite`)

CLI で確認する場合は次のように開けます。

```bash
sqlite3 "$(xcrun simctl get_app_container booted com.basematch.app data)/Documents/base_match_local.sqlite"
```

よく見るテーブルは次の 3 つです。

```sql
SELECT * FROM local_games;
SELECT * FROM local_players;
SELECT * FROM local_plate_appearances;
```

## ローカル開発環境のセットアップ

### 1. Supabase CLIのインストール

```bash
# macOS (Homebrew)
brew install supabase/tap/supabase

# その他のOS: https://supabase.com/docs/guides/cli
```

### 2. ローカルSupabaseの起動

```bash
# プロジェクトルートで実行
supabase start
```

起動後、以下の情報が表示されます:
- API URL: `http://127.0.0.1:54321`
- anon key: ローカル用の匿名キー
- service_role key: ローカル用のサービスロールキー
- Studio URL: `http://127.0.0.1:54323`（管理画面）

### 3. 環境変数ファイルの作成

プロジェクトルートに`.env.local`を作成:

```bash
# .env.local（ローカル開発用）
# ※ URLは常に 127.0.0.1 で記載してください
# Androidエミュレータの場合はアプリが自動的に 10.0.2.2 に変換します
SUPABASE_URL=http://127.0.0.1:54321
SUPABASE_ANON_KEY=<supabase startで表示されたanon key>
```

### 4. ローカルSupabaseの停止

```bash
supabase stop

# データを保持したまま停止（次回起動時に状態を復元）
supabase stop --backup

# データをリセットして停止
supabase stop --no-backup
```

## Supabase CLI コマンドリファレンス

### ローカル環境の操作

```bash
# ローカル環境の状態確認（URL・各種キーを再表示）
supabase status

# ローカルDBを初期状態に戻す（マイグレーション＋シードを再適用）
supabase db reset

# Dockerイメージなど関連リソースを最新化
supabase update
```

### プロジェクトの初期化・リンク

```bash
# 新規プロジェクトで supabase/ ディレクトリを作成
supabase init

# ローカルプロジェクトをSupabase Cloud上のプロジェクトに紐付け
supabase link --project-ref <your-project-ref>

# ログイン（初回のみ、ブラウザが開いてトークンを発行）
supabase login

# ログアウト
supabase logout

# リンク中のプロジェクト一覧
supabase projects list
```

### マイグレーション

```bash
# 新しいマイグレーションファイルを作成
supabase migration new <マイグレーション名>

# 適用済みマイグレーション一覧の確認（ローカル／リモート）
supabase migration list

# ローカルDBの差分から自動でマイグレーションファイルを生成
supabase db diff -f <ファイル名>

# マイグレーションをローカルに適用
supabase migration up

# マイグレーションをリモート（本番）に反映
supabase db push

# リモートDBの現状をローカルに取り込む
supabase db pull
```

### 型定義の生成（Dart/TypeScript等）

```bash
# ローカルDBからTypeScript型定義を出力
supabase gen types typescript --local > types/supabase.ts

# リモートDBから型定義を出力
supabase gen types typescript --project-id <your-project-ref> > types/supabase.ts
```

※ Flutter/Dartでは型自動生成は未対応のため、`freezed` + 手書きDTOで対応します。

### シードデータ

```bash
# supabase/seed.sql を再実行（db reset 時にも自動実行される）
supabase db reset
```

### Edge Functions

```bash
# 新規Edge Function作成
supabase functions new <関数名>

# ローカルで関数を実行
supabase functions serve <関数名>

# リモートにデプロイ
supabase functions deploy <関数名>

# 関数の削除
supabase functions delete <関数名>
```

### シークレット管理

```bash
# 環境変数（シークレット）をリモートに設定
supabase secrets set KEY=value

# 現在のシークレット一覧
supabase secrets list

# シークレットの削除
supabase secrets unset KEY
```

### ログ・デバッグ

```bash
# DB・API・Authなどのログを表示
supabase logs db
supabase logs api
supabase logs auth

# ヘルプ表示
supabase --help
supabase <command> --help
```

## リリース環境のセットアップ

### 1. Supabase Cloudプロジェクトの作成

[Supabase Dashboard](https://supabase.com/dashboard)でプロジェクトを作成します。

### 2. 環境変数ファイルの作成

プロジェクトルートに`.env`を作成:

```bash
# .env（リリース用）
SUPABASE_URL=https://<your-project-ref>.supabase.co
SUPABASE_ANON_KEY=<Supabase Dashboardで取得したanon key>
```

## Flutterでの環境変数の読み込み

### 1. パッケージの追加

```yaml
# pubspec.yaml
dependencies:
  flutter_dotenv: ^5.1.0
```

### 2. アセットへの登録

```yaml
# pubspec.yaml
flutter:
  assets:
    - .env
    - .env.local
```

### 3. コードでの使用例

```dart
import 'package:flutter_dotenv/flutter_dotenv.dart';

Future<void> main() async {
  // 開発環境では.env.local、リリースでは.envを読み込む
  const isDebug = bool.fromEnvironment('dart.vm.product') == false;
  await dotenv.load(fileName: isDebug ? '.env.local' : '.env');

  final supabaseUrl = dotenv.env['SUPABASE_URL']!;
  final supabaseAnonKey = dotenv.env['SUPABASE_ANON_KEY']!;

  // Supabaseの初期化
  // ...
}
```

## 環境を切り替えて実行する

### dart-defineを使用した環境切り替え

`--dart-define`オプションを使用して、実行時に環境を明示的に指定できます。

```bash
# ローカル開発環境で実行
flutter run --dart-define=ENV=local

# リリース環境で実行
flutter run --dart-define=ENV=release
```

### ビルド時の環境指定

```bash
# ローカル環境向けビルド（Android）
flutter build apk --dart-define=ENV=local

# リリース環境向けビルド（Android）
flutter build apk --dart-define=ENV=release

# リリース環境向けビルド（iOS）
flutter build ios --dart-define=ENV=release
```

### コードでの環境判定

`--dart-define`で指定した値をコード内で取得して環境を切り替えます:

```dart
import 'dart:io';
import 'package:flutter_dotenv/flutter_dotenv.dart';

Future<void> main() async {
  // --dart-define=ENV=xxx で指定した値を取得（デフォルトは'local'）
  const env = String.fromEnvironment('ENV', defaultValue: 'local');

  // 環境に応じた.envファイルを読み込む
  final envFile = env == 'release' ? '.env' : '.env.local';
  await dotenv.load(fileName: envFile);

  var supabaseUrl = dotenv.env['SUPABASE_URL']!;
  final supabaseAnonKey = dotenv.env['SUPABASE_ANON_KEY']!;

  // ローカル開発時、Androidエミュレータでは 127.0.0.1 → 10.0.2.2 に変換
  if (env == 'local' && Platform.isAndroid) {
    supabaseUrl = supabaseUrl.replaceAll('127.0.0.1', '10.0.2.2');
  }

  // Supabaseの初期化
  // ...
}
```

## リリース環境へのデータベースマイグレーション

ローカルで作成したデータベーススキーマ（`supabase/migrations/` 内のSQLファイル）を、Supabase Cloudの本番データベースに適用する手順です。

### 1. Supabase CLIで本番プロジェクトにリンク

```bash
supabase link --project-ref <your-project-ref>
```

`project-ref`はSupabase DashboardのURLから確認できます:
```
https://supabase.com/dashboard/project/xxxxxxxx
                                        ^^^^^^^^ ← これがproject-ref
```

実行するとデータベースパスワードの入力を求められます。プロジェクト作成時に設定したパスワードを入力してください。

### 2. 適用内容の事前確認（任意）

```bash
supabase db push --dry-run
```

`--dry-run`オプションを付けると、実際には実行せず適用されるマイグレーションの一覧を確認できます。

### 3. マイグレーションの実行

```bash
supabase db push
```

`supabase/migrations/`内のSQLファイルが番号順に本番データベースへ適用されます。

### 4. スキーマの差分確認

```bash
supabase db diff
```

ローカルと本番のスキーマに差分がある場合に表示されます。

### 新しいマイグレーションの追加

テーブルの追加・変更が必要な場合:

```bash
# 新しいマイグレーションファイルを作成
supabase migration new <変更内容の名前>

# → supabase/migrations/2026XXXX_<変更内容の名前>.sql が生成される
# → 中にSQLを記述
# → supabase db push で本番に適用
```

## APIキーについて（2025年11月以降の新プロジェクト）

2025年11月以降に作成されたSupabaseプロジェクトでは、従来の`anon key`/`service_role key`が廃止され、新しいキー体系に変更されています。

| 旧キー | 新キー | 用途 |
|--------|--------|------|
| `anon key` | **Publishable key**（`sb_publishable_...`） | クライアント側（Flutter等）で使用 |
| `service_role key` | **Secret key**（`sb_secret_...`） | サーバー側で使用（公開しない） |

Publishable keyは従来のanon keyと同じ使い方ができるため、`.env`の`SUPABASE_ANON_KEY`にPublishable keyを設定すれば、コード変更は不要です。

キーの取得場所: Supabase Dashboard → Settings → API Keys

## Supabase CLIコマンド一覧

ローカル開発でよく使うコマンドをまとめています。すべてプロジェクトルートで実行してください。

### ローカル環境の起動・停止

| コマンド | 説明 |
|---------|------|
| `supabase start` | ローカルSupabase環境をDockerで起動する。初回はイメージのダウンロードで時間がかかる。起動後にAPI URL、anon key、Studio URLなどが表示される |
| `supabase stop` | ローカル環境を停止する。データは保持される |
| `supabase stop --no-backup` | ローカル環境を停止し、データベースのボリュームも削除する（完全リセットしたい場合） |
| `supabase status` | 現在のローカル環境の状態（起動中かどうか）とAPI URL・キー情報を表示する |

### データベース操作

| コマンド | 説明 |
|---------|------|
| `supabase db reset` | ローカルDBを完全にリセットし、全マイグレーションを再適用してから`seed.sql`を実行する。スキーマを壊してしまったときや、まっさらな状態に戻したいときに使う |
| `supabase db diff` | ローカルDBの現在の状態と`migrations/`フォルダの内容を比較して、差分のSQLを出力する。Studio画面でテーブルを直接変更した後、その変更をマイグレーションファイルに落とし込むときに便利 |
| `supabase db diff --schema public` | publicスキーマだけに絞って差分を確認する |
| `supabase db push` | `migrations/`内のマイグレーションファイルをリモート（本番）のSupabaseプロジェクトに適用する。`supabase link`済みであることが前提 |
| `supabase db push --dry-run` | 実際には適用せず、どのマイグレーションが適用されるかを確認する |
| `supabase db lint` | スキーマに対してリンター（セキュリティやパフォーマンスの観点）を実行する |

### マイグレーション

| コマンド | 説明 |
|---------|------|
| `supabase migration new <名前>` | 新しいマイグレーションファイルを作成する。`supabase/migrations/`に`YYYYMMDDHHMMSS_<名前>.sql`というファイルが生成されるので、中にSQLを書く |
| `supabase migration list` | ローカルとリモートのマイグレーション適用状況を一覧表示する。どこまで適用済みかを確認できる |
| `supabase migration up` | まだ適用されていないマイグレーションをローカルDBに適用する（`db reset`と違い、既存データは保持される） |
| `supabase migration repair --status applied <バージョン>` | マイグレーションの適用状態を手動で修正する。本番DBで手動でSQLを実行した場合などに使う |

### プロジェクト連携

| コマンド | 説明 |
|---------|------|
| `supabase link --project-ref <ref>` | ローカルプロジェクトをSupabase Cloudのプロジェクトに紐付ける。`db push`や`functions deploy`の前に必要。DBパスワードの入力を求められる |
| `supabase projects list` | 自分のSupabaseアカウントに紐付いたプロジェクトの一覧を表示する |

### Edge Functions（サーバーレス関数）

| コマンド | 説明 |
|---------|------|
| `supabase functions new <名前>` | 新しいEdge Function（Deno/TypeScript）のテンプレートを`supabase/functions/<名前>/`に作成する |
| `supabase functions serve` | ローカルでEdge Functionsを起動する。ホットリロード対応なので、コード変更が即座に反映される |
| `supabase functions deploy <名前>` | 指定したEdge Functionを本番にデプロイする |
| `supabase functions deploy` | すべてのEdge Functionsを本番にデプロイする |

### 型生成

| コマンド | 説明 |
|---------|------|
| `supabase gen types typescript --local` | ローカルDBのスキーマからTypeScriptの型定義を自動生成する。Edge Functionsで型安全にDBアクセスしたい場合に使う |
| `supabase gen types typescript --project-id <ref>` | リモート（本番）のスキーマから型定義を生成する |

### その他

| コマンド | 説明 |
|---------|------|
| `supabase login` | Supabase CLIにログインする。`link`や`deploy`などリモート操作の前に必要 |
| `supabase --version` | CLIのバージョンを確認する |
| `supabase update` | CLIを最新バージョンに更新する（Homebrewの場合は`brew upgrade supabase`） |
| `supabase inspect db` | データベースの統計情報（テーブルサイズ、インデックス使用状況など）を確認する |

### よくある使い方の流れ

```bash
# 1. ローカル環境を起動
supabase start

# 2. 開発中にテーブル変更が必要になったら
supabase migration new add_some_column
# → 生成されたSQLファイルにALTER TABLEなどを記述

# 3. マイグレーションを適用（ローカル）
supabase db reset   # 全リセット＆再適用（シードデータも入る）
# または
supabase migration up   # 未適用分だけ適用（既存データは残る）

# 4. 本番に反映
supabase link --project-ref <ref>   # 初回のみ
supabase db push --dry-run          # 確認
supabase db push                    # 適用

# 5. 開発終了
supabase stop
```

## 注意事項

- `.env`および`.env.local`は`.gitignore`に追加済みのため、Gitにコミットされません
- 環境変数ファイルのサンプルとして`.env.example`を参照してください
- ローカル開発時は必ず`supabase start`でローカル環境を起動してから開発を行ってください
- `.env.local`の`SUPABASE_URL`は常に`http://127.0.0.1:54321`で記載してください。Androidエミュレータで実行する場合は、アプリが自動的に`10.0.2.2`に変換します（エミュレータからホストPCにアクセスするため）
