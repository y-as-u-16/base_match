## 1. 事前準備・方針確定

1. **Phase 1 の前提確認**
   - Flutter 単体でローカル記録版をリリースできるようにする。
   - Supabase 非依存でも成立する画面導線と保存方式を確定する。
2. **Phase 2 以降の接続方針**
   - `supabase` プロジェクト（ローカル or リモート）を確定。
   - すべてのDDLは **Supabase migration** で管理する。
3. **スキーマ命名/規約**
   - UUIDは `gen_random_uuid()` を使用。
   - テーブル/カラムは設計書の名称に一致させる。

## 2. DB実装（Phase 2 以降）

### 2.1 マイグレーション設計（フェーズ分割）

**フェーズ0: ローカルモード基盤**
- 端末内ストレージの選定
- ローカル試合/打席/個人成績モデルの作成
- ローカルデータからクラウドデータへ移行する変換方針を確定

**フェーズ1: 認証基盤・ユーザー関連**
- `users` テーブル作成
- 主要インデックス追加（`firebase_uid` unique）

**フェーズ2: チーム関連**
- `teams` 作成
- `team_members` 作成（`team_id`/`user_id` FK）
- `(team_id, user_id)` の unique 制約

**フェーズ3: プレイヤーと試合**
- `players` 作成（`team_id`/`user_id`/`created_by` FK）
- `games` 作成（`home_team_id`/`away_team_id` FK）

**フェーズ4: 打席イベント/Claim/監査ログ**
- `plate_appearances` 作成（`game_id`/`batter_player_id`/`pitcher_player_id` FK）
- `claims` 作成（`player_id`/`claimed_user_id` FK）
- `audit_logs` 作成（FKは `actor_user_id` のみ）

**フェーズ5: 集計ビュー**
- `v_matchup_batter_pitcher`
- `v_matchup_team_team`

**フェーズ6: RLSポリシー**
- 参照/更新ポリシーをテーブル単位で実装
- 段階的に「参照のみ」→「更新/削除」へ拡張

**フェーズ7: 公開閲覧用データ**
- 共有済み因縁カードの公開用テーブル/ビュー
- 匿名閲覧可能な最小データだけを投影

---

### 2.2 マイグレーション詳細（例）

1. **初期マイグレーション**
   - `users` 作成、`firebase_uid` に unique。
2. **チーム・メンバー**
   - `teams` → `team_members` の順で作成。
3. **プレイヤー/試合**
   - `players`（仮プレイヤー対応）
   - `games`（`status` default `draft`）
4. **打席イベント/Claim/監査ログ**
   - `plate_appearances`、`claims`、`audit_logs` を追加。
5. **ビュー作成**
   - 集計定義（`AB`, `H`, `HR` 等）を SQL View として実装。
6. **RLS/インデックス**
   - RLS: `teams/team_members/games/plate_appearances/players/claims` を対象。
   - 主要参照カラム（`game_id`, `batter_player_id`, `pitcher_player_id`）に索引付与。
7. **公開閲覧投影**
   - 共有済み因縁だけを別投影で公開。
   - 元のゲーム/打席データは匿名公開しない。

## 3. アプリ実装（MVP機能の段階的実装）

### 3.1 ローカルモード（M1）
1. 未ログインで試合作成、打席入力、個人成績表示まで完結できるようにする。
2. 端末内ストレージに保存する Repository を用意する。
3. ローカルデータ件数を確認できる UI を用意する。

### 3.2 認証・クラウド同期（M2）
1. Supabase Auth（Google / Apple）導入。
2. `users` テーブルと連動したプロフィール作成/更新。
3. ローカル記録をクラウドへ移行するフローを追加。

### 3.3 試合管理（M2）
1. ログイン後はクラウド側で試合作成（Draft）。
2. 試合一覧/詳細取得。
3. 確定（Final）への状態変更。

### 3.4 打席イベント入力（M1/M2）
1. 打席入力画面（MVP簡易）。
2. `plate_appearances` 登録。
3. 結果コードの選択UI（`result_type`, `result_detail`）。

### 3.5 個人成績表示（M1/M2）
1. 未ログイン時はローカル集計で個人成績表示。
2. ログイン後は `v_matchup_batter_pitcher` を参照。
3. 直近N試合はクエリで取得。

### 3.6 チーム管理（M3 / 課金）
1. チーム作成画面・API。
2. `team_members` 追加（Owner/Admin/Member）。
3. 招待コード生成/参加フロー。

### 3.7 因縁カード（M3 / 課金）
1. 因縁詳細画面。
2. カード生成（Canvas描画）。
3. 共有機能（Share Sheet）。
4. 共有済みカードの公開閲覧画面。

### 3.8 仮プレイヤー・Claim（M4）
1. 仮プレイヤー作成（`players.user_id is null`）。
2. Claim UI 追加 → `claims` 登録。
3. Claim後のプロフィール統合。

## 4. RLSポリシー実装手順

1. **参照ポリシー（最小）**
   - 所属チーム/試合参加者のみ閲覧。
   - 公開閲覧は別投影データのみ匿名許可。
2. **更新/削除ポリシー**
   - `Owner/Admin` のみ編集可。
   - `plate_appearances` は試合作成者 or Admin のみ変更。
3. **監査ログ**
   - 更新系APIで必ず `audit_logs` に記録。

## 5. テスト・検証

1. **DB検証**
   - マイグレーション実行順に `supabase db reset` で再構築。
   - Seedデータを投入して基本クエリを確認。
2. **ユースケース検証**
   - 未ログイン → 試合作成 → 打席入力 → 個人成績表示までを通し確認。
   - ログイン → ローカル記録移行 → クラウド保存までを通し確認。
   - 課金ユーザー → チーム参加 → 因縁表示/共有までを通し確認。
3. **RLS検証**
   - チーム未所属ユーザーでアクセス拒否を確認。
   - 公開共有URLだけ匿名閲覧できることを確認。

---

## 6. 実装マイルストーン（基本設計との対応）

- **M1**：ローカル記録 + 打席入力 + 個人成績表示
- **M2**：認証 + クラウド同期 + 試合CRUD
- **M3**：課金導線 + チーム + 因縁詳細 + カード生成 + 共有
- **M4**：仮プレイヤー + Claim
- **M5**：βテスト → 改善

---

## 7. 実装完了後の運用

1. **DBマイグレーション運用**
   - 変更は必ず migration で履歴管理。
2. **データ再集計**
   - 必要に応じて `plate_appearances` から集計ビュー再構築。
3. **将来拡張**
   - Edge Functions/AIテキスト生成、詳細スタッツ、通知機構。
