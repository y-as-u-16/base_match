# 04_ローカル記録版へのリプレイス計画

## 1. 目的

本ドキュメントは、現在の「ログインしないと何も使えない」構成から、Phase 1 の「未ログインでローカル記録のみ使える」構成へアプリをリプレイスするための計画を整理したものです。

対象は以下の状態変化です。

- 現状:
  - Supabase 初期化が起動時に必須
  - 認証済みでないと主要画面へ進めない
  - Repository が Supabase 前提
- 目標:
  - Supabase なしでもアプリが起動する
  - 未ログインで試合作成、打席入力、個人成績表示まで完結する
  - Phase 2 で Google / Apple ログインとクラウド同期を追加できる構成にする

## 2. 現状の制約

現在の主要な制約点は以下です。

1. 起動が Supabase 初期化に依存している
   - [lib/features/splash/presentation/pages/splash_page.dart](/Users/egiyasuyuki/dev/base_match/lib/features/splash/presentation/pages/splash_page.dart:34)
   - `dotenv.load()` と `Supabase.initialize()` が完了しないと先へ進まない

2. ルーターが認証を前提に全画面をガードしている
   - [lib/core/routing/app_router.dart](/Users/egiyasuyuki/dev/base_match/lib/core/routing/app_router.dart:108)
   - `authStateProvider` を見て、未認証なら `/login` に強制遷移する

3. 認証方式が Email/Password 前提になっている
   - [lib/features/auth/presentation/view_models/auth_view_model.dart](/Users/egiyasuyuki/dev/base_match/lib/features/auth/presentation/view_models/auth_view_model.dart:26)
   - [lib/features/auth/data/repositories/auth_repository_impl.dart](/Users/egiyasuyuki/dev/base_match/lib/features/auth/data/repositories/auth_repository_impl.dart:15)

4. データ取得・保存が Supabase Client 前提になっている
   - [lib/core/network/supabase_client_provider.dart](/Users/egiyasuyuki/dev/base_match/lib/core/network/supabase_client_provider.dart:4)
   - `games`, `matchups`, `profile`, `teams` などの Repository 実装が `SupabaseClient` に依存している

## 3. リプレイス方針

Phase 1 では「Supabase を消す」のではなく、「Supabase を optional にして、ローカル実装を先に通す」方針を取る。

この方針の理由:

- 既存 feature 構造を崩さずに差し替えやすい
- Phase 2 でクラウド実装を戻しやすい
- Domain 層の interface を維持しながら、Data 層だけをローカル実装へ置換できる

基本戦略は以下です。

1. 起動時の必須依存から Supabase を外す
2. ルーターを「未ログインでも使える前提」に変える
3. Phase 1 で必要な機能だけ、ローカル Repository 実装を追加する
4. 課金・チーム・因縁は画面または導線ごと一時的に閉じる
5. Phase 2 向けに、ログイン後のクラウド移行ポイントを残す

## 4. 目標アーキテクチャ

### 4.1 アプリ状態

Phase 1 では認証状態ではなく、利用モードを基準にする。

- `local_only`
  - 未ログイン
  - ローカル保存
  - 試合作成、打席入力、個人成績表示のみ
- `cloud_enabled`
  - Phase 2 以降で追加
  - Google / Apple ログイン済み
  - Supabase 保存と移行が有効

### 4.2 Repository 方針

Phase 1 時点では次の二層を明示する。

- `Local*RepositoryImpl`
  - Phase 1 の実体
  - 端末内ストレージを使う
- `Supabase*RepositoryImpl`
  - Phase 2 以降で使う
  - 現行実装をベースに維持

切り替えは Riverpod Provider で行う。

## 6. 実施順

### Step 1. 起動を Supabase 非依存にする

目的:

- `.env.local` や Supabase がなくてもアプリを起動できる状態にする

作業:

- `SplashPage` の初期化を分離
- `isInitializedProvider` の意味を「Supabase 初期化完了」から「アプリ初期化完了」に変更

完了条件:

- `flutter run` で Supabase 未起動でもホームへ入れる

### Step 2. ルーターの認証ガードを撤去する

目的:

- ログインしなくても主要導線へ進めるようにする

作業:

- `authStateProvider` 前提の redirect を外す
- 未ログインでも使える公開ルートだけを残す

完了条件:

- `/login` を経由せずホーム、試合作成、打席入力へ進める

### Step 3. ローカル保存の Repository を追加する

目的:

- Phase 1 のコアデータを端末内保存できるようにする

作業:

- ローカル保存用 model / mapper / repository を追加
- 現行の `GameRepository` interface を生かしたまま provider で差し替える

完了条件:

- アプリ再起動後も試合と打席が保持される

### Step 4. 個人成績表示をローカル集計へ切り替える

目的:

- Supabase view なしでも Phase 1 の価値を成立させる

作業:

- batter / pitcher の簡易集計ロジックをクライアント側で実装
- Phase 1 は個人成績に限定し、因縁一覧は出さない

完了条件:

- 打席入力結果が個人成績へ反映される

### Step 5. Phase 1 外機能を閉じる

目的:

- チーム、因縁、プロフィール編集などの未対応機能で壊れないようにする

作業:

- タブ構成を整理
- 未提供機能は非表示または「今後追加」画面に置換

完了条件:

- Phase 1 の公開機能だけでアプリ全体が破綻なく回る

### Step 6. Phase 2 の差し込みポイントを残す

目的:

- 後から Google / Apple ログインと Supabase 同期を追加しやすくする

作業:

- `AppMode` または `StorageMode` のような状態を用意
- local repository と cloud repository の切り替え点を provider に集約
- ローカル記録を export しやすい形に整える

完了条件:

- Phase 2 で認証を追加しても大規模再設計にならない

## 7. リスク

1. 既存 UI がチーム/因縁タブ前提で組まれている
   - タブ設計の見直しが必要

2. `GameRepository` だけで足りない可能性がある
   - 打席、プレイヤー、成績集計の責務分割を確認する必要がある

3. 現在の `matchups` 画面は Supabase view 前提
   - Phase 1 では無理に維持しない方がよい

4. 将来のクラウド移行時にローカルIDとサーバーIDの対応が必要
   - local UUID を先に採用しておくと移行しやすい

## 8. 推奨リプレイス単位

一気に全部置き換えず、以下の単位で進める。

1. 起動とルーティング
2. ローカル保存の導入
3. 試合作成と打席入力
4. 個人成績表示
5. Phase 1 外機能の整理

この順に進める理由:

- まず「ログインしないと何もできない」を解消できる
- 次にデータ保存を成立させられる
- その後で UI の不足を埋められる

## 9. Phase 2 への接続

Phase 2 で追加する内容:

- Google / Apple ログイン
- Supabase 初期化
- ローカル記録のクラウド移行
- Claim

Phase 1 時点で残しておくべき前提:

- Domain interface は極力維持する
- local entity は cloud entity と対応する形にする
- ローカル記録の一覧取得と一括 export を可能にする

## 10. 完了の定義

Phase 1 のリプレイス完了は、以下を満たした状態とする。

- Supabase 未設定でもアプリが起動する
- ログインなしで試合作成できる
- ログインなしで打席入力できる
- ログインなしで個人成績表示ができる
- アプリ再起動後もローカル記録が残る
- チーム、因縁、課金前提機能が壊れずに隠されている
